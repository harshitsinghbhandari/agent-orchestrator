ahead Branch Feature Analysis

Issue #87: Code-Atlas — Project-Specific Long-Term Memory

Goal: Give each orchestrated coding agent a structured, persistent semantic index of the target codebase — a "living atlas" that is built, updated, and queried automatically.

Technical design:

  • A .atlas/ directory committed to the repo root contains:
    • modules.json — LLM-annotated module/package semantic summaries
    • patterns.json — architectural patterns, conventions, anti-patterns
    • decisions.log — append-only record of agent decisions and rationale
    • hotspots.json — files/functions with high churn or interaction frequency
  • Atlas is incrementally updated via GitHub Actions workflow on merge to main
  • Agents query via ao context [tags] before beginning work

Issue #86: Skills Compounding System

Goal: When an agent successfully solves a non-trivial coding task, the solution strategy is distilled into a reusable "skill."

Mechanics:

  • Post-task reflection: orchestrator asks "what general approach solved this?" and stores in .atlas/skills/
  • Similarity matching: before spawning, query skills store for analogous past tasks
  • Quality scoring: skills upvoted/downvoted based on subsequent success; poor skills decay and are pruned
  • Cross-project skills: high-quality skills can be published to public atlas-hub
The Compound Effect

The 10th task in a domain is dramatically faster and more accurate than the 1st, because 9 prior skills have been accumulated. This is the defining self-improving narrative that will drive virality.

Features to Add

Strategic Constraint

Add only 2–3 features that create immediate "wow" moments, build directly on Code-atlas and Skills Compounding, and can ship within 4 weeks. Every addition must justify its complexity with measurable user delight.

Feature 01

Atlas Dashboard — Live Memory Visualization

A real-time web dashboard that visualizes the Code-atlas as it grows — showing which modules have been mapped, which skills have been accumulated, which architectural patterns have been discovered.

  • Module map: Force-directed graph of annotated modules (D3.js)
  • Skills timeline: Scrolling feed with success rate badges
  • Decision log: Searchable log linkable to PRs
  • Atlas health gauge: Coverage percentage and staleness indicator

Effort: 1.5 weeks | Impact: ~40% of demo viewers share a screenshot

Feature 02

Atlas Export + "Share Your Atlas" Social Layer

Let teams export anonymized, sanitized atlas snapshots and share them publicly on atlas-hub.io.

  • ao atlas export --sanitize — strips code, preserves patterns
  • atlas-hub.io — public index searchable by tech stack
  • ao atlas import <url> — bootstrap from similar projects
  • Each export includes "Built with agent-orchestrator ahead" badge

Effort: 1 week | Impact: 200+ atlases shared in first month

Feature 03

"Replay Mode" — Agent Decision Playback

Play back any completed agent session step-by-step in the dashboard — seeing each tool call, atlas query, skill lookup, and code change as an annotated narrative.

  • Timeline slider + code diff panel + reasoning annotations
  • ao replay <session-id> --share generates shareable HTML
  • Replays become "show don't tell" marketing assets

Effort: 2 weeks | Impact: ~15% of users share a replay in first week

The Seamlessness Principle

A user's first 3 minutes determine whether they star the repo, share it, or close the tab. Every second of friction in those 3 minutes costs approximately 8% of potential stars.

Error Handling: The No-Blame Policy

Every error must: state what happened in plain English, suggest the most likely fix with a copy-pasteable command, link to docs, and never blame the user.

Chapter 05

Problems the User Must Never Face

Security Guardrails

Risk Required Mitigation
Agent writes outside worktree Path containment: .worktreeinclude allowlist; any write outside kills session and logs violation
Secret exposure in atlas Atlas build scans for secrets using gitleaks; block commit if detected
Prompt injection via issue text Sanitize all issue body content; strip HTML/markdown that could inject instructions
Runaway agent cost Hard token budget per session; alert and pause at 80% consumed
Fork bomb / infinite spawn Max concurrent agent limit; detect and kill fork-bomb patterns
Atlas data poisoning Append-only writes with version hash; modifications require re-verification

Reliability Requirements

  • Atlas staleness protection: CI prevents agents from running on stale atlas (>24h); auto-rebuild before spawning
  • Session recovery: ao recover restores crashed session state (worktree, PR branch, current task)
  • Idempotent atlas builds: Running twice produces same result (fixed seed, temperature=0)
  • Atomic skill writes: SQLite transactions; no corrupt records on crash
  • Dashboard offline mode: Load historical data when orchestrator not running (read-only from SQLite)
