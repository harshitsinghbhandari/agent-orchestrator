01 — Foundation

What is OpenUI?

A full-stack Generative UI framework. LLMs output a declarative language instead of JSON or HTML. The runtime handles all interactivity without calling the LLM again.

LLM OutputOpenUI Lang
LexerToken stream
ParserStreaming AST
MaterializerResolve refs
Evaluator$vars, Query()
RendererReact Components

Key Insight

Generate once, execute forever. The LLM produces a declarative program with Query(), Mutation(), $variables, and Action(). The runtime handles data fetching, form interactions, and re-rendering with zero additional LLM calls.

defineComponent()

02 — Opportunities

Integration Opportunities

Four focused integrations, scoped to where OpenUI genuinely outperforms static components. Copilot Panel and Multi-Agent Comparison are deferred — complexity-to-value ratio is wrong until the basic pipeline is proven.

01 Agent Insight Cards + PR Review Briefing
High ImpactMedium
▸
Agents produce rich, interactive summaries of their work at session completion. Decision logs, file change maps, test results, cost metrics, risk assessment, and review attention markers — all in one card, surfaced in both SessionDetail and PR view.

Cost model: One LLM call per completed session, cached in metadata forever. No recurring cost.

Why OpenUI fits here specifically: Agent output is inherently variable — different issues produce different decisions, structures, trade-offs. A static template either over-provisions (empty sections) or under-provisions (no section for unexpected work). The "generate once" model is a feature here, not a limitation — completed sessions don't change.

02 Session Timeline (Completed Sessions Only)
Medium ImpactMedium
▸
Constraint: OpenUI has no push/SSE support. A timeline for an active session is stale the moment it renders. This only works for completed sessions.

For active sessions: use a static Tailwind timeline component driven by lifecycle events. For completed sessions: generate an annotated, interactive timeline where the LLM adds narrative context ("tried approach A, failed because X, pivoted to B").

Cost model: One LLM call per completed session, cached.

03 On-Demand Analytics
Medium ImpactEasy
▸
Honest assessment: This is the weakest OpenUI use case. Analytics data is structured and predictable — a chart library (Recharts, Nivo) handles 90% of it at 10% of the complexity without an LLM call.

The one scenario where OpenUI adds value: natural-language ad-hoc queries where the LLM correlates data with contextual explanations ("why did failure rates spike Tuesday?").

Recommendation: Build basic analytics with a chart library first. Only reach for OpenUI if the natural-language query UX proves valuable beyond what structured charts provide.

05 — Risk Mitigations

Risk Mitigations

Four risks identified in review. Each has a concrete mitigation plan.

Invalid OpenUI Lang

LLM may generate malformed code. Mitigation: React error boundary around every <Renderer>. Falls back to existing plain-text summary. Validate generated code before storing in metadata. The insight card is a progressive enhancement — if it fails, the user sees what they'd see today.

Security / Trust Boundary

Renderer evaluates generated code. Mitigation: OpenUI Lang is declarative, not JS — no arbitrary code execution. Component library is the sandbox. Query/Mutation allowlisted to /api/* only. CSP headers block external resources. Residual risk is content trust (misleading summaries), same as plain text today.

Bundle Size

Lexer + parser + evaluator + Zod v4. Budget: 50KB gzipped max (15% increase). Measure with next build + bundle analyzer after adding dep. If over budget: dynamic import with loading skeleton. If over 100KB: reconsider the feature entirely.

Zod v4 Coexistence

Two Zod versions in one monorepo. Plan: Isolate v4 to a single file (openui-components.ts). CI grep check blocks v4 imports elsewhere. No v4 types in public interfaces. Add zod@4 to web package only, not root. Document in CLAUDE.md.

Degradation Invariant

Every code path that displays an insight card must have a non-OpenUI fallback. The degradation chain: OpenUI renders → error boundary catches → plain-text summary → "No summary available". The dashboard never depends on OpenUI being functional.

07 — Start Here

Recommended Starting Point

Phase 1: Agent Insight Cards. Tests the core thesis where OpenUI's value is clearest — variable agent output that static templates can't handle.

Why start with Insight Cards, not Analytics?

Analytics data is structured and predictable — a chart library handles it at a fraction of the complexity. Insight Cards test the actual thesis: LLM-generated UI handles variable agent output (decisions, trade-offs, file changes) better than static templates. Each completed session produces unique output. One LLM call, cached forever. The "generate once" model is a feature here, not a limitation.
