Quick Summary

What this is: Treat Ollama as a CLI-based agent by wrapping the ollama run <model> command, reusing the existing tmux runtime.

Core Concept

Treat Ollama exactly like Claude CLI or Codex CLI - spawn as subprocess, pipe stdin/stdout.

Key Benefit

Minimal code - reuses existing tmux runtime entirely.

Complexity

Low

Single new agent plugin

How Users Will Use It

  • Config: agent: ollama in project config
  • Launch: ollama run llama3 spawned as subprocess
  • Prompt: Sent via tmux send-keys to stdin
  • Output: Captured via tmux capture-pane from stdout
Important: This is the simplest approach but has significant UX limitations: no streaming, no conversation history, no tool calling.

Architecture Overview

Architecture Diagram

Key Insight

Ollama is NOT an agent here. It's a subprocess that happens to be an LLM. The agent plugin is minimal - it just tells the orchestrator how to spawn and interact with the ollama process.

How It Works

Step-by-Step Flow

1. Launch

Agent plugin returns launch command:

getLaunchCommand() {
  const model = "llama3";
  return { command: "ollama", args: ["run", model] };
}

Result: Process spawned via tmux

2. Prompt

Messages sent via stdin:

// User sends message
sendMessage("Hello, how are you?");

// TMUX sends to process stdin
tmux send-keys -t session "Hello, how are you?" Enter

Result: Process receives prompt via stdin

3. Output

Response captured from stdout:

// Read output from tmux pane
const output = await tmux(
  "capture-pane",
  "-t", session,
  "-p"
);

// Return to orchestrator
return output;

Result: Full response captured after process exits

Process Lifecycle

  • Spawn: Creates tmux session running ollama run <model>
  • Active: Process is running (even if idle waiting for input)
  • Exit: Process exits when user sends /exit or Ctrl+D
  • Destroy: tmux session killed via kill-session

Agent Plugin Implementation

import {
  shellEscape,
  Agent,
  AgentLaunchConfig,
  ActivityDetection,
  PluginModule
} from "@composio/ao-core";

# =============================================================================
# Plugin Manifest
# =============================================================================

export const manifest = {
  name: "ollama",
  slot: "agent" as const,
  description: "Agent plugin: Ollama CLI",
  version: "0.1.0",
};

# =============================================================================
# Agent Implementation
# =============================================================================

function createOllamaAgent(): Agent {
  return {
    name: "ollama",
    processName: "ollama",
    promptDelivery: "inline",  # Prompt goes in launch command

    getLaunchCommand(config: AgentLaunchConfig): string {
      const model = config.ollamaOptions?.model ?? "llama3";

      # Build command with model and optional parameters
      const parts: string[] = ["ollama", "run", shellEscape(model)];

      if (config.ollamaOptions?.temperature) {
        parts.push("--temperature", String(config.ollamaOptions.temperature));
      }

      if (config.ollamaOptions?.numCtx) {
        parts.push("--numCtx", String(config.ollamaOptions.numCtx));
      }

      if (config.prompt) {
        parts.push(shellEscape(config.prompt));
      }

      return parts.join(" ");
    },

    getEnvironment(config: AgentLaunchConfig): Record<string, string> {
      return {
        AO_SESSION_ID: config.sessionId
      };
    },

    async detectActivity(terminalOutput: string): Promise<ActivityState> {
      # If output ends with prompt character, idle
      if (terminalOutput.trim().endsWith(">> ") || terminalOutput.trim().endsWith("$ ")) {
        return "idle";
      }
      return "active";
    },

    async getActivityState(session, readyThresholdMs): Promise<ActivityDetection> {
      const processRunning = await this.isProcessRunning(session.runtimeHandle);
      if (!processRunning) {
        return { state: "exited" };
      }
      # Cannot distinguish "running" from "waiting for input"
      # Always return active or idle based on last activity time
      return { state: "active" };
    },

    async isProcessRunning(handle): Promise<boolean> {
      # Use tmux to check if session exists
      try {
        await execFileAsync("tmux", ["has-session", "-t", handle.id]);
        return true;
      } catch {
        return false;
      }
    },

    async getSessionInfo(session): Promise<AgentSessionInfo> {
      return {
        summary: null,  # No summary from CLI
        agentSessionId: null,
        cost: { inputTokens: 0, outputTokens: 0, estimatedCostUsd: 0 }
      };
    }
  };
}

export function create(): Agent {
  return createOllamaAgent();
}

export function detect(): boolean {
  try {
    await execFileAsync("ollama", ["--version"]);
    return true;
  } catch {
    return false;
  }
}

export default { manifest, create, detect } satisfies PluginModule<Agent>;

Key Decisions

  • promptDelivery: "inline": Prompt goes in launch command, not sent post-launch
  • No streaming: Must wait for full response before capture
  • Simple activity detection: Cannot distinguish "thinking" from "waiting"

Limitations

⚠️ Important Limitations

These limitations are inherent to the CLI-wrapper approach. Users should understand these tradeoffs.

1. No Streaming

Problem: Cannot stream tokens as they generate

  • Full response only available after process exits
  • User waits for entire generation before seeing any output
  • No progressive rendering like Claude/Codex

Why: tmux captures only after session ends

2. No Conversation History

Problem: Ollama CLI is stateless between prompts

  • Each prompt is a new, independent request
  • Must manually prepend all previous messages
  • Token limits become problematic quickly
  • No actual "session" context

Why: CLI doesn't maintain conversation state

3. No Tool Calling

Problem: Ollama CLI doesn't support function calling natively

  • Tool calling is an API feature, not CLI feature
  • Cannot leverage Ollama's function calling capabilities
  • Users miss out on powerful tool use patterns

Why: CLI interface doesn't expose tool calling

4. Poor Activity Detection

Problem: Cannot distinguish "thinking" from "waiting"

  • Process appears "running" even when idle
  • No way to detect if agent is "ready for input"
  • Lifecycle manager sees constant "active" state

Why: No native activity indicators in CLI output

5. No Real-time Output

Problem: Cannot show progressive generations

  • User sees blank tmux session while model generates
  • No character-by-character streaming display
  • Wait time feels longer (no visual feedback)

When to Choose Option B

✅ Good For:

  • Quick proof-of-concept
  • Development/testing (not production)
  • Users who prefer CLI interaction
  • Environments where HTTP runtime seems overkill

When NOT to Choose Option B

❌ Avoid If:

  • You need streaming for UX
  • You need tool calling capabilities
  • You need conversation history
  • You want Ollama as a true "agent"

FAQ

Q: Can Ollama run as a service?

A: Yes, but for Option B, the CLI wrapper spawns ollama run <model> as a subprocess. Ollama itself can run as a background service, but that's separate from the agent plugin. The plugin focuses on the CLI command interaction.

Q: Does this require tmux?

A: Yes. The current implementation uses the tmux runtime to manage the CLI session. This provides consistent behavior across all agents. (Process runtime could be used too, but tmux gives better output capture.)

Q: What happens when prompt exceeds token limit?

A: The CLI returns an error. The orchestrator captures it and marks the session as "errored". Users must then reduce prompt size or use a model with larger context window.

Q: Can I use this with remote Ollama?

A: Only if the remote Ollama CLI is installed on the same machine. For remote Ollama, Option A or C (HTTP backend) is more appropriate. This CLI wrapper assumes local Ollama installation.

Q: How do I specify which Ollama model to use?

A: In your project config:

projects:
  myproject:
    agent: ollama
    agentConfig:
      model: llama3:8b-instruction
Q: Is this production-ready?

A: For simple use cases, yes. But with limitations: no streaming, no tool calling, no history. If production needs these features, consider Option A or C instead.
