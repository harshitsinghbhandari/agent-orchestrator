Quick Summary

What this is: A new HTTP-based runtime plugin that treats Ollama as a first-class agent by communicating via REST API instead of spawning CLI processes.

Core Concept

Replace process spawning with HTTP requests to Ollama's `/api/chat` endpoint.

Key Benefit

Ollama becomes equally first-class as Claude, Codex, and other agents.

Complexity

High

New runtime pattern + new agent plugin

What Changes for Users?

  • Config changes: Users select runtime: http for Ollama sessions
  • Launch command: No shell command needed - HTTP server handles everything
  • Activity detection: Timing-based (last message age) instead of process state
  • Session management: Full control via API - no tmux/process cleanup needed
TL;DR: Build a new HTTP runtime that talks to Ollama's API, plus a small agent plugin to adapt Ollama to the agent interface. This gives Ollama first-class status with proper streaming, tool calling, and session history.

Two New Plugins

1. Runtime Plugin (runtime-http)

What it is: A generic HTTP runtime that manages HTTP connections to any HTTP-based agent server.

interface HttpRuntimeConfig {
  baseUrl: string;           // http://localhost:11434
  model?: string;             // default model
  keepAlive?: string;         // "5m", etc.
  timeout?: number;          // request timeout
}

export function create(config?: HttpRuntimeConfig): Runtime {
  return {
    name: "http",
    async create(handle) { /* return handle with connection info */ },
    async destroy(handle) { /* cleanup */ },
    async sendMessage(handle, message) {
      // POST to /api/chat with streaming
      // Store response chunks for getOutput()
    },
    async getOutput(handle, lines) { /* return accumulated output */ },
    async isAlive(handle) { /* ping /api/version */ },

}

Implementation Details

HTTP Runtime - Key Design Decisions

1. State Management

Challenge: Ollama is stateless - we must manage conversation history client-side.

Solution: Store message history in session metadata or runtime handle data.

// Runtime handle data stores history
interface HttpHandleData {
  messages: Message[];
  lastActivityAt: number;
  model: string;
  baseUrl: string;
  sessionId: string;
}

2. Streaming Support

How it works:

  1. User sends message via sendMessage()
  2. HTTP POST to /api/chat with stream: true
  3. Read streaming JSON lines from response
  4. Store each chunk in runtime's output buffer
  5. Return accumulated output via getOutput()
Tip: The streaming logic can be shared with other HTTP-based agents (e.g., OpenAI, Groq) by making the HTTP runtime generic.

3. Activity Detection

Challenge: No process to check - Ollama server is stateless.

Solution: Timing-based detection

getActivityState(session, readyThresholdMs = 300000) {
  if (!session.runtimeHandle) return { state: "exited" };

  const lastActivity = session.runtimeHandle.data.lastActivityAt;
  const age = new Date().getTime() - lastActivity;

  if (age > readyThresholdMs) {
    return { state: "idle", timestamp: new Date(lastActivity) };
  }

  return { state: "ready", timestamp: new Date(lastActivity) };
}

4. Error Handling

Key errors to handle:

  • Connection refused: Ollama server not running
  • Model not found: Invalid model name configured
  • Timeout: Request took too long
  • Rate limited: Ollama returns 429

Recovery strategy: Gracefully fail, log error, let user retry

Integration Points

Session Manager Changes

When spawning a session with Ollama agent + HTTP runtime:

  1. Create HTTP runtime handle with base URL and empty message history
  2. Launch Ollama server if not running (optional)
  3. Send initial prompt via sendMessage()
  4. Store session metadata with runtime handle ID

Lifecycle Manager Changes

Check Ollama sessions via polling:

  1. Get session with Ollama agent + HTTP runtime
  2. Call agent.getActivityState(session)
  3. Update session status based on activity
  4. If idle too long, mark as "needs_input" or "idle"

Making the Foundation Strong

1. Abstract the HTTP Client

Create a shared HTTP client module that can be used by multiple HTTP-based agents:

# packages/plugins/runtime-http/src/http-client.ts

export interface HttpClientConfig {
  baseUrl: string;
  timeout: number;
  headers?: Record<string, string>;
}

export class HttpClient {
  private config: HttpClientConfig;

  constructor(config: HttpClientConfig) {
    this.config = config;
  }

  async chat(messages: Message[], options?: ChatOptions): Promise<ChatResponse> {
    // POST to /api/chat with streaming support
  }

  async health(): Promise<HealthResponse> {
    // GET /api/version or /health
  }
}
Benefit: HTTP runtime becomes reusable for any HTTP-based agent (Ollama, OpenAI, custom API servers).

2. Session Metadata Integration

Store HTTP session state in the same metadata system:

# Session metadata format
interface HttpSessionMetadata {
  runtimeType: "http";
  baseUrl: string;
  messages: Message[];  # JSON array
  lastActivityAt: string;  # ISO timestamp
  stats: {
    requestCount: number;
    totalTokens: number;
  };
}

3. Error Handling Strategy

Connection Errors

  • Detect: Catch network errors, connection refused
  • Log: Include URL, attempt count, retry timing
  • Notify: Mark session as "errored" with clear message
  • Recover: User must fix server or restart Ollama

Request Errors

  • 400 Bad Request: Invalid model, malformed JSON
  • 401 Unauthorized: API key missing/invalid
  • 429 Too Many Requests: Rate limited - add backoff retry
  • 500 Server Error: Ollama crash - mark session as errored

4. Testing Strategy

Unit Tests

  • sendMessage() - basic message
  • sendMessage() - long message
  • sendMessage() - streaming response
  • getOutput() - empty buffer
  • getOutput() - accumulated output
  • isAlive() - healthy server
  • isAlive() - server down

Integration Tests

  • Full agent lifecycle with Ollama server
  • Session spawn → message → response
  • Multiple consecutive messages
  • Session restore after crash

5. Configuration Validation

Add validation in the runtime create() function:

export function create(config?: HttpRuntimeConfig): Runtime {
  const resolvedConfig = {
    baseUrl: config?.baseUrl ?? "http://localhost:11434",
    model: config?.model ?? "llama3",
    timeout: config?.timeout ?? 60000,
    keepAlive: config?.keepAlive ?? "5m",
  };

  // Validate URL format
  try {
    new URL(resolvedConfig.baseUrl);
  } catch (err) {
    throw new Error(Error.message);
  }

  // Validate keepAlive format (e.g., "5m", "1h")
  if (resolvedConfig.keepAlive && !/^[0-9]+[smhd]$/.test(resolvedConfig.keepAlive)) {
    throw new Error(Error.message);
  }

  // ... return runtime
}

Tradeoffs & Considerations

Pros

✅ Clean Separation

Ollama is treated equally with other agents. No CLI-specific hacks needed.

✅ Full Streaming Support

Native HTTP streaming via Ollama's /api/chat endpoint.

✅ Tool Calling & JSON Mode

Ollama's native features accessible via API parameters.

✅ Client-Side History

Full control over conversation history management.

✅ No Process Cleanup

No need to kill child processes or tmux sessions.

✅ Generic Runtime

HTTP runtime can be reused for other API-based agents.

Cons

❌ Implementation Effort

Largest implementation effort among options. Requires new runtime + new agent plugin.

❌ New Runtime Pattern

Diverges from existing CLI-based model. Teams must learn new pattern.

❌ Maintenance burden

HTTP connection management, error handling, retry logic.

❌ Activity Detection

Timing-based instead of process-based. Less precise.

FAQ - Common Questions

Q: Why not use the "process" runtime with Ollama CLI?

A: You could! But Ollama's CLI has limitations:

  • The -p flag causes one-shot mode (agent exits after response)
  • No interactive session for follow-up questions
  • Less control over conversation history
  • Shell escaping issues with long prompts
Q: Does this require Ollama to be running?

A: Yes. But the agent plugin can auto-start Ollama:

getLaunchCommand(config) {
  if (!isOllamaRunning()) {
    startOllamaServer();
  }
  return "";  // No command needed - server runs in background
}
Q: How is this different from "process" runtime with HTTP server?

A: Good question! The "process" runtime would:

  • Start Ollama as a child process
  • Use sendMessage() to send to stdin
  • Parse stdout for response

The "http" runtime:

  • Starts Ollama as separate process
  • Connects via HTTP to /api/chat
  • Reads JSON from streaming response

HTTP is cleaner and more reliable.

Q: What happens if Ollama server crashes?

A: The HTTP client returns an error. The lifecycle manager:

  1. Detects error state
  2. Updates session status to "errored"
  3. Notifies user (if notifier configured)
  4. Can optionally restart Ollama
Q: Can multiple agents share the same Ollama instance?

A: Yes! All agents connect to the same HTTP server (localhost:11434). Each session maintains its own message history.

Q: How do I configure Ollama models in my project?

A: Add to your agent-orchestrator.yaml:

projects:
  myproject:
    repo: ...
    agent: ollama
    agentConfig:
      model: llama3:8b-instruction
      temperature: 0.7
      numCtx: 8192
Q: What about tool calling? Does Ollama support it?

A: Yes! Ollama supports tool calling via the API. The HTTP runtime passes tool definitions and captures tool responses:

# HTTP request
{
  model: "llama3",
  messages: [...],
  tools: [
    {
      type: "function",
      function: {
        name: "get_weather",
        parameters: {...}
      }
    }
  ]
}
Q: Is this production-ready?

A: This is a design proposal. Production readiness depends on:

  • Test coverage
  • Error handling in production scenarios
  • Performance under load
  • Monitoring and observability
Q: What about security? Are credentials exposed?

A: Ollama runs locally by default. For remote Ollama instances:

  • Use HTTPS for encrypted communication
  • Store API keys in environment variables
  • Network isolation (localhost only)
