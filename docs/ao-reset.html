PR #1380 · feat/ao-reset-command

ao reset — what it does, how it works

A maintenance command that wipes a project's local Agent Orchestrator state — disk, registry, preferences, event log — refuses to do so while the orchestrator is live, prints a loud red destructive warning before the prompt, and never touches a persistence layer it doesn't need to. Plus the bugs caught in review, including a silent file side-effect on fresh machines, and how they were fixed.

What it does

AO accumulates per-project state across four disjoint persistence layers. Before ao reset, recovering a corrupted project meant manual rm -rf, hand-editing config.yaml, and surgery on preferences.json. Reset bundles that into one command.

Layer Path What's there Reset wipes?
V2 storage ~/.agent-orchestrator/projects/<projectId>/ Sessions, worktrees, feedback reports, orchestrator runtime state, agent JSONL logs Yes
Global registry ~/.agent-orchestrator/config.yaml The project's entry + its slot in projectOrder Yes
Portfolio prefs ~/.agent-orchestrator/portfolio/preferences.json Pinned/enabled state, displayName, projectOrder, defaultProjectId Yes
Activity events ~/.agent-orchestrator/activity-events.db SQLite rows keyed by project_id Yes (per-project DELETE)
The repo on disk Wherever it lives Your code + agent-orchestrator.yaml Never
Legacy V1 storage ~/.agent-orchestrator/<storageKey>/ Pre-V2 sessions/worktrees if migrated No (V2 migration's job)
Observability ~/.agent-orchestrator/<hash>-observability/ Process snapshots / audit logs (shared across projects of one config) No (shared key)

Surface

ao reset                       # auto-detect from cwd (single-project or matching cwd)
ao reset <project>             # explicit project id
ao reset -p <project>          # same, via flag
ao reset --all                 # every configured project
ao reset --yes                 # skip the confirmation prompt

Conflicting selectors are rejected up-front: --all with a positional or -p; the positional and --project with different values.

Execution flow

0
Reject contradictory flags. Before doing any work, refuse incompatible selector combinations with a non-zero exit.
1
Load config. loadConfig() returns LoadedConfig; we resolve targets via explicit arg, single-project, or findProjectForDirectory(cwd).
2
Live-orchestrator guard. getRunning() reads running.json. If ao start is serving any of the targeted projects, refuse with a clear "stop first" hint and exit 1. Without this, the dashboard's 5-second poll would race with our rmSync and corrupt in-memory orchestrator state.
3
Preview. For each target, call collectDeletionTargets(projectId) to compute { baseDir, exists, items }. Print a per-target listing of what will be deleted, including whether the global registry / preferences will also be touched. — packages/cli/src/commands/reset.ts
4
Confirm. Interactive callers (isHumanCaller()) get a promptConfirm with default false. Non-interactive callers must pass --yes; otherwise exit 1.
5
Kill live sessions. SessionManager.list(projectId) → SessionManager.kill(id) per session. kill() destroys the runtime (tmux/process/etc.), removes the AO-managed worktree, deletes any agent-native session record, and writes a terminated lifecycle entry. Per-session failures are swallowed — a dead session shouldn't block the wipe.
6
Disk wipe. If baseDir exists, rmSync(baseDir, { recursive: true, force: true }). ENOENT is treated as success (preview→remove race). Other errors are recorded as a per-target failure.
7
Global state cleanup (best-effort). unregisterProject(projectId) removes the registry entry and prunes projectOrder. updatePreferences() drops the per-project slot, removes the projectId from preferences.projectOrder, and clears defaultProjectId if it pointed at the removed project.
8
Activity events (best-effort). deleteEventsForProject(projectId) runs DELETE FROM activity_events WHERE project_id = ? against the shared SQLite log.
9
Honest exit. Aggregate per-target results. If any disk wipe failed, print a summary and exit 1. Otherwise print ✓ Reset complete. and exit 0.

Safety guards

Don't reset under a live ao start

The orchestrator polls the V2 storage dir on a 5-second cadence. Wiping files under it produces EBUSY errors, half-killed sessions, and inconsistent dashboard state. The fix is "stop first," not "make the daemon resilient to the rug being pulled." Reset reads running.json via getRunning(); if the targeted project appears in running.projects, we refuse with a clear hint and exit 1.

Confirmation in interactive contexts only

promptConfirm only runs for isHumanCaller(). Non-human callers (CI, agent spawn scripts) must pass --yes. Symmetric with the rest of the CLI: machines don't give consent implicitly to destructive operations.

Honest exit codes

Per-target outcomes are tracked. Any disk wipe failure produces a non-zero exit and a summary block. Shell wrappers (CI pipelines, outer agents) treat exit code as the authoritative signal — printing ✓ Reset complete while exiting 0 after a silent failure was the original bug this rework fixed.

Bugs caught in review & how they were fixed

Two passes of review (/pr-review and Copilot bot) found issues that all keyed off the same kind of mistake: treating a preview list as the source of truth for whether something exists. Fixes were grouped into two pushes.

Bug · core Reset claimed success without removing the directory.

packages/cli/src/commands/reset.ts:393 — disk removal only ran when items.length > 0, but items came from a best-effort readdirSync. An existing-but-empty or unreadable directory produced an empty items array, so rmSync never ran — yet the final line still printed ✓ Reset complete.

packages/cli/src/commands/reset.ts:315 — the same proxy fed the "Nothing to reset" early-exit, so the same condition could short-circuit the entire command.

Fix · a68e9ccb Track baseDir existence as a first-class flag.

collectDeletionTargets now returns { baseDir, exists, items }. Both the early-exit predicate and the disk-removal branch key off exists: 

// before:
if (allTargets.every(t => t.items.length === 0 && !registered(t))) return;

if (items.length > 0) rmSync(baseDir, ...);

// after:
if (allTargets.every(t => !t.exists && !registered(t))) return;

if (exists) {
  try { rmSync(baseDir, { recursive: true, force: true }); }
  catch (err) {
    if (err.code === "ENOENT") diskRemoved = true;  // race: dir vanished
    else                       diskError   = err.message;
  }
}

Added a regression test ("removes an empty baseDir when it exists but has no listable contents") that creates an empty project dir and verifies it gets removed and the output contains Reset complete — not No AO state found.

Fix · ec7d8201 Honest exit codes.

The original code printed ✓ Reset complete regardless of per-target outcome and exited 0. The fix tracks each target's disk result, prints a failure summary listing each failed project, and exits 1 on any failure.

Fix · ec7d8201 Live-orchestrator guard.

Reset now refuses while ao start is serving the targeted project. The guard reads running.json through getRunning() and intersects running.projects with the resolved targets. If any overlap, we print which projects are affected, suggest ao stop <project>, and exit 1.

Fix · ec7d8201 Extended scope to all four persistence layers.

The original PR only touched the V2 storage dir. After review, reset also unregisters the project from the global config registry, prunes its slot from portfolio preferences (including defaultProjectId and projectOrder), and runs a per-project DELETE against the activity-events SQLite log. A new core helper deleteEventsForProject(projectId) keeps the SQLite logic in events-db.ts rather than leaking SQL into the CLI.

Polish · a68e9ccb Runtime-agnostic wording.

The docstring and CLI doc said "Live tmux sessions are killed first", but SessionManager.kill is runtime-agnostic — it works for tmux, the process runtime, and any future runtime plugin. Reworded to "Live sessions are killed first via the configured runtime."

Polish · a68e9ccb Don't claim atomicity that doesn't exist.

The implementation doc said reset cleans state "atomically per project." It doesn't — global config, prefs, and SQLite cleanup are intentionally best-effort. Disk wipe can succeed independently of global-state cleanup. Replaced with an explicit non-transactional disclaimer; the exit code reflects only the disk-wipe outcome.

Bug · ux Reset created an empty preferences.json as a side effect.

Caught by the user on a fresh machine: ~/.agent-orchestrator/portfolio/preferences.json didn't exist before reset, but did after. The original pruneGlobalState called updatePreferences(), which always writes back regardless of whether the updater modified anything. On a machine that's never customized the portfolio, this spawned a stub { "version": 1 } file out of nowhere.

Reporting was also imprecise: the post-removal line said "Unregistered from global config + portfolio preferences" even when prefs.json didn't exist. And the preview's "+ global registry entry + portfolio preferences" hint was only printed when the project was in the global registry, missing the case of a project that's prefs-referenced but not registered.

Fix · latest Per-layer accuracy + no spurious writes.

Replaced updatePreferences with explicit loadPreferences + change-detection + savePreferences: 

// Only write if the prefs file actually references this project.
// Otherwise we'd materialize an empty preferences.json on a fresh machine.
const prefs = loadPreferences();
if (preferencesReferenceProject(prefs, projectId)) {
  // remove the matching keys...
  savePreferences(prefs);
  prefsChanged = true;
}
return { unregistered, prefsChanged };

Each layer reports independently. Preview now shows "+ entry in global config registry" and "+ slot in portfolio preferences" on separate lines, only when each is actually present. Post-removal output uses "Unregistered from global config registry" and "Pruned slot from portfolio preferences" as separate lines — each printed only when that layer actually changed.

A new readGlobalReferences(projectId) helper does the pre-execution check so the preview is always accurate. Two new tests pin the behavior: "does NOT create preferences.json when no prefs reference the project" and "does NOT touch preferences.json when prefs exist but reference other projects".

Fix · latest Loud red destructive-action warnings.

Before the preview, reset now prints a bold red banner: 

  ⚠  WARNING — DESTRUCTIVE OPERATION
  This permanently deletes project state. It cannot be undone.
  Make sure you have pushed any work in active worktrees first.

After the per-target list, a "boundaries" notice tells the user what reset doesn't touch — the most common source of confusion: 

  NOT touched by reset:
    • The repo on disk and your agent-orchestrator.yaml
    • Legacy V1 storage (~/.agent-orchestrator/<storageKey>/)
    • The shared observability dir (~/.agent-orchestrator/<hash>-observability/)

The confirmation prompt itself is now bold red. The live-orchestrator-detected refusal got an upgraded banner too: "⚠ CANNOT RESET — `ao start` IS LIVE" with the affected projects listed underneath.

Fix · ec7d8201 Test isolation via vi.mock("node:os").

The original test suite tried to redirect ~/.agent-orchestrator/ by setting process.env.HOME. This silently didn't work under Vitest — libuv's uv_os_homedir didn't honor the per-test HOME mutation, so tests were touching the user's real ~/.agent-orchestrator/projects/. The pollution went unnoticed because reset's own rmSync cleaned up after itself.

Replaced with a proper vi.mock("node:os", ...) that redirects homedir() to a per-test tmpdir. Now isolation is real, not accidental.

Test coverage

22 tests covering: empty-baseDir regression, per-confirmation removal, preview output, no-state path, kill-then-wipe ordering, confirmation denial, --all, unknown project, -p flag, kill-failure tolerance, non-TTY refusal, live-ao start guard (refuse + proceed branches), --all + project-arg rejection, conflicting positional/-p rejection, rmSync failure exit code, registry+prefs pruning, activity events pruning, --all on empty config, no-prefs-write when file is missing, no-prefs-write when prefs reference other projects, and destructive-warning banner is rendered.

The biggest find of the review wasn't a bug in production code — it was that the tests had been quietly running against the user's real ~/.agent-orchestrator/ for the entire life of the PR. The visible failure (chmod ENOENT on a path that "should" have existed) was the symptom that exposed it.
