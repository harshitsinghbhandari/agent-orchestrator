Quick Summary

What this is: Treat Ollama as a model backend rather than an agent. Instead of spawning Ollama as a runnable agent, expose it as a local model option that other agents can use for inference.

Core Concept

Ollama is infrastructure (like OpenAI), not an agent. Claude uses Ollama for inference, but still runs as Claude.

Key Benefit

Full streaming, tool calling, and conversation history - all from existing agents using Ollama as backend.

Complexity

Medium

New model provider + abstraction

How Users Will Use It

  • Configure: User sets modelProvider: ollama when creating agents
  • Use: Agents (Claude, etc.) use Ollama for inference without spawning as separate agent
  • Result: All agent features work + local models available
TL;DR: Ollama becomes a "model" option like OpenAI/Anthropic. You still chat with Claude - Claude just uses Ollama under the hood.

Architecture Overview

Key Insight

Ollama is NOT an agent here. It is a model provider - infrastructure that agents use for inference, just like OpenAI, Anthropic, or Groq.

Architecture Diagram

How It Differs

Option A (HTTP Runtime)

Ollama is an agent. User launches Ollama directly. Ollama manages its own session history.

Option B (CLI Wrapper)

Ollama is an agent. User launches Ollama CLI. No session history.

Option C (Local Backend)

Ollama is a model provider. User launches Claude (or other agent), which uses Ollama for inference.

How It Works

Step-by-Step Flow

1. Ollama Server Runs

Ollama runs as a service (not managed by orchestrator):

# User starts Ollama separately
ollama serve

# Or install as systemd service
systemctl enable ollama
systemctl start ollama

2. Model Provider Registered

Ollama appears as a model option alongside OpenAI, Anthropic:

# User configures agent with Ollama as backend
projects:
  myproject:
    agent: claude
    agentConfig:
      modelProvider: ollama
      model: llama3:8b-instruction

3. Agent Uses Ollama for Inference

Claude agent sends prompts to Ollama API:

# Agent sends to Ollama API
POST http://localhost:11434/api/chat
Content-Type: application/json

{
  model: "llama3:8b-instruction",
  messages: [
    { role: "user", content: "Hello" }
  ],
  stream: true
}

4. Streaming Response

Response streamed back to agent:

HTTP/1.1 200 OK
Transfer-Encoding: chunked

data: {"message": {"content": "Hello"}, "done": false}
data: {"message": {"content": " there"}, "done": false}
data: {"message": {"content": "!"}, "done": true}

Model Provider Interface

ModelProvider Interface

What model providers implement:

interface ModelProvider {
  name: string;

  # Synchronous completion (no streaming)
  async complete(
    prompt: string,
    options?: ModelOptions
  ): Promise<string>;

  # Streaming completion
  async *completeStream(
    prompt: string,
    options?: ModelOptions
  ): AsyncIterable<string>;

  # List available models
  async listModels(): Promise<string[]>;

  # Check if provider is available
  async isAvailable(): Promise<boolean>;
}

Ollama Provider Implementation

import { ModelProvider, ModelConfig } from "@composio/ao-core";

export interface OllamaConfig extends ModelConfig {
  baseUrl: string;         # default: http://localhost:11434
  model: string;                 # e.g., "llama3", "codellama"
  keepAlive?: string;            # "5m", etc.
  temperature?: number;
  numCtx?: number;
}

export function createOllamaProvider(config: OllamaConfig): ModelProvider {
  return {
    name: "ollama",

    async complete(prompt: string, options: Record<string, unknown> = {}): Promise<string> {
      const response = await fetch("${config.baseUrl}/api/chat", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          model: config.model,
          messages: [{ role: "user", content: prompt }],
          stream: false,
          options: {
            temperature: options.temperature ?? config.temperature,
            num_ctx: options.numCtx ?? config.numCtx,
          },
        }),
      });

      const data = await response.json();
      return data.message.content;
    },

    async *completeStream(prompt: string, options: Record<string, unknown> = {}): AsyncIterable<string> {
      const response = await fetch("${config.baseUrl}/api/chat", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          model: config.model,
          messages: [{ role: "user", content: prompt }],
          stream: true,
          options: {
            temperature: options.temperature ?? config.temperature,
            num_ctx: options.numCtx ?? config.numCtx,
          },
        }),
      });

      const reader = response.body.getReader();
      const decoder = new TextDecoder();

      while (true) {
        const { done, value } = await reader.read();
        if (done) break;

        const chunk = decoder.decode(value);
        for (const line of chunk.split("\n")) {
          if (line.startsWith("{")) {
            const data = JSON.parse(line);
            if (data.message?.content) {
              yield data.message.content;
            }
          }
        }
      }
    },

    async listModels(): Promise<string[]> {
      const response = await fetch("${config.baseUrl}/api/tags");
      const data = await response.json();
      return data.models.map((m: any) => m.name);
    },

    async isAvailable(): Promise<boolean> {
      try {
        const response = await fetch("${config.baseUrl}/api/version");
        return response.ok;
      } catch {
        return false;
      }
    }
  };
}

    FAQ

    Q: How is this different from Option A?

    A: Option A creates Ollama as an agent with a custom HTTP runtime. Option C creates Ollama as a model provider that existing agents use for inference.

    • Option A: User launches launchAgent("ollama") - Ollama is the agent
    • Option C: User launches launchAgent("claude", {model: "ollama"}) - Claude is the agent, Ollama is the backend
    Q: Do I need to run Ollama as a service?

    A: Yes. Ollama must be running as ollama serve before using Option C. The model provider doesn't manage the Ollama server lifecycle - it just connects to it.

    Users can:

    • Install Ollama as a system service
    • Run ollama serve manually
    • Use Docker: docker run -p 11434:11434 ollama/ollama
    Q: Can multiple agents share the same Ollama instance?

    A: Yes. All agents connect to the same Ollama server (localhost:11434). Each agent session maintains its own conversation history and state - Ollama is stateless.

    Q: Does this require changing existing agents?

    A: Yes, slightly. Existing agents (Claude, Codex, etc.) need to support the modelProvider option to use Ollama. This is a modest change to agent implementations.

    Q: What happens if Ollama server crashes?

    A: The model provider returns an error. The agent catches the error and can retry or fail gracefully. The orchestrator can notify the user.

    Q: Is there a performance difference vs Option A?

    A: Performance is similar - both use HTTP to Ollama API. Option C may be slightly more efficient since the model provider abstraction is designed for this use case from the start.

    Q: Can I use this with remote Ollama?

    A: Yes! Just configure the baseUrl to point to your remote Ollama instance. This is a key advantage - local or remote, the API is the same.
